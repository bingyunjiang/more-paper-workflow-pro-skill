# Step 1: 交互式确定研究主题（v2.0 增强版）

> 旧版 Step 1 仅靠 5 轮问答让用户自述研究方向。v2.0 在对话中注入外部信息（文献量/趋势/gap/创新性），让用户做有依据的决策。

---

## 1. 启动前读取 (Pre-read Checklist)

执行本步骤前，必须确认以下文件已加载：

- [ ] `.skill-state/error_log.md` — 已知错误及修复规则
- [ ] `.skill-state/decision_log.md` — 影响本 Step 的结构性决策

---

## 2. 适用任务 (Applicable Tasks)

- 新用户首次使用，需要厘清研究方向
- 已有大致方向但需要精确到具体选题
- 选题可行性评估（文献饱和度、gap 识别、创新点预判）
- 研究阶段诊断（硕士/博士/青年教师 → 推荐最优入口）

---

## 3. 不适用任务 (Non-applicable Tasks)

以下任务不属于本 Step 范围：

- 已有清晰选题和文献 → 可直接跳到 Step 3（检索方案）或 Step 5（下载）
- 大纲生成 → 路由到 `agents/step_2_outline.md`
- 文献检索 → 路由到 `agents/step_4_search_score.md`

> **硬边界：** Step 1 只生成问题结构，不生成章节结构。一级/二级标题、章节顺序、章节证据需求表完整版都属于 Step 2。

---

### 5.1. 决策包式交互规则 (Decision Packages)

Step 1 的交互目标不是“一次只问一个问题”，而是**一轮只收敛一个决策包**。

默认按以下 3 个决策包推进：

1. `研究对象 + 应用场景`
2. `核心问题 + 方法路径`
3. `可行性 + 创新点边界`

执行要求：

- 每个决策包内部可以包含 2-4 个强相关小问题。
- 一轮结束必须给出一个中间结论，不能只继续追问。
- 不要一轮同时推进多个互不相关的不确定性。
- 当信息足够进入 Step 2 时，主动停止澄清并切换到产出模式。

中间结论至少应包括以下之一：
- 当前聚焦对象
- 当前问题定义
- 候选方向取舍
- 是否进入 Step 2

## 6. 执行流程 (Execution Flow)

### 1.0. 增强架构总览

```
Step 1.1: 研究阶段诊断      → 用户画像 + 推荐入口 + research_intent_type
Step 1.2: 广度探索 + 预检索  → 发散子方向 → 文献量/趋势/gap初判 + Socratic 澄清
Step 1.3: 深度聚焦          → 研究问题候选 + 方法论深化 + 创新点预判
Step 1.4: 选题预审          → originality/importance/feasibility + 证据支撑评分 + 反方挑战
                              🟢绿灯 / 🟡黄灯 / 🔴红灯
Step 1.5: 检索深度推断      → quick|standard|deep + 参数理由 + 方法蓝图交接

产出: 研究主题.md（增强版）
  含结构化元数据 + 用户画像 + 预检索证据 + 研究问题候选 + scope 边界 + 方法蓝图 + 创新点初判 + 预审结论 + Step 2/3 handoff
```

### 1.1. 研究阶段诊断

> **目的：** 先了解用户是谁、处于什么阶段，再决定走哪条路。不是所有人都需要完整的 8 步流程。

**诊断维度：**

| 诊断项 | 选项 | 推荐入口 |
|--------|------|---------|
| **身份** | 硕士生 / 博士生 / 研究者 / 工程师 | — |
| **年级** | 博一（找方向）/ 博二（开题）/ 博三+（冲刺） | — |
| **当前阶段** | 刚入学，需要导师给方向 | → 先泛读综述，再走 Step 1-8 |
| | 已有大致方向，需精确选题 | → 完整走 Step 1-8 |
| | 已有具体题目，需文献支撑 | → 跳 Step 3（检索方案） |
| | 已有大纲，需下载文献写论文 | → 跳 Step 5（下载） |
| | 已有初稿，需润色 | → 跳 Step 8（润色） |
| **时间压力** | 3 个月内要投稿 / 6 个月 / 无紧迫 | 影响检索年限和下载策略 |
| **目标期刊** | 已有目标 / 不确定 / 毕业就行 | 影响写作和润色标准 |

**若用户方向极模糊**（如“新能源”“医学影像”“AI+教育”），不要先追问细节。先进入主动发散模式：

| 发散轴 | 输出要求 |
|--------|----------|
| 应用场景 | 列出 3-5 个可落地场景 |
| 方法路线 | 列出可做综述/仿真/实验/算法/系统实现的路线 |
| 数据或实验条件 | 判断公开数据、实验平台、仿真工具或问卷访谈是否可得 |
| 论文策略 | 给出快毕业型、稳妥开题型、高创新投稿型三类路线 |
| 风险提示 | 说明每条路线的文献支撑、创新拥挤度和执行成本 |

**Socratic 澄清输出要求：**

当用户输入属于“兴趣/模糊方向”而不是明确研究问题时，Step 1 必须在 1.1-1.3 之间逐步逼出以下结构化结果，而不是直接跳到大纲：

- `research_intent_type`：探索型 / 明确研究型 / 综述型 / 事实核查型 / 系统综述型
- `research_question_candidates`：1-3 个候选研究问题
- `scope_boundaries`
  - `in_scope`
  - `out_of_scope`
- `sub_questions`：2-3 个子问题

若用户一开始就给出清晰研究问题，可压缩该过程，直接输出结构化 RQ，不强制长对话。

### 1.2. 广度探索 + 预检索

> **目的：** 用户说出一个大致方向后，先做一次快速文献扫描——让数据说话。

**流程：**

```
用户说"充电桩热管理" →
  ① 发散 3-5 个子方向：
      ├── 冷板式液冷散热
      ├── 相变材料（PCM）热缓冲
      ├── 充电枪接口发热
      ├── 功率模块 IGBT 温升
      └── 整柜风道优化
  ② 逐个子方向快速预检索（Semantic Scholar / Crossref）
  ③ 趋势判断：
      500+ 篇 → ⚠️ 热门但拥挤，需要更窄的切入点
      50-200 篇 → ✅ 适中，有足够文献又有创新空间
      <20 篇 → ⚠️ 太新/太小众，文献支撑不足
  ④ 输出对比表，让用户选择 1-2 个进入深度聚焦
```

**预检索执行协议：**

| 项目 | 要求 |
|------|------|
| 子方向数量 | 3-5 个，必须覆盖不同对象/方法/场景，不要只是同义改写 |
| 关键词组 | 每个子方向至少 2 组关键词：中文方向需中英双语，英文方向至少包含同义词/缩写 |
| 数据源 | 优先 Crossref/OpenAlex/Semantic Scholar；中文方向可补 CNKI/万方入口提示 |
| 检索规模 | 每个子方向快速查看前 20-50 条结果或可用总量估计 |
| 时间窗口 | 默认关注近 3 年趋势，同时保留经典/综述线索 |
| 证据记录 | 至少记录代表性论文/综述 2-3 篇、近三年活跃度、主要方法、初判 gap |
| 降级策略 | 外部检索失败时，明确标注“需 Step 4 复核”，不得伪造文献量 |

**子方向对比表模板：**

| 子方向 | 关键词组 | 文献量级 | 近三年趋势 | 代表论文/综述 | 初判 gap | 风险 | 建议 |
|--------|----------|----------|------------|---------------|----------|------|------|
| 方向 A | keyword set | <20 / 50-200 / 200-500 / 500+ | 上升/平稳/下降/未知 | 2-3 条 | 1-2 条 | 拥挤/过窄/条件不足 | 保留/收窄/放弃 |

**文献量级解释：**

| 文献量级 | 判断 | 动作 |
|----------|------|------|
| 500+ | 热门但拥挤 | 必须收窄对象、场景、方法或指标 |
| 200-500 | 文献充足但竞争明显 | 需要明确差异化切入 |
| 50-200 | 较理想 | 可进入 1.3 深度聚焦 |
| 20-50 | 可探索但支撑偏弱 | 需要扩大同义词或上位概念 |
| <20 | 太新/太窄/检索词不准 | 先扩展概念，不建议直接定题 |

### 1.3. 深度聚焦

> **目的：** 进入深度聚焦——不只问"做什么"，还预判"创新点在哪"。

**双循环结构：**

```
Cycle 1: 方法论深化
  从"仿真" → 深化为 "CFD 参数化建模 + 代理模型优化 + 实验验证"
  → 方法越具体，后续检索方案越精准

Cycle 2: 创新点预判
  基于该方向的 gap 初判，与用户讨论可能的创新切入：
  ├── 方法创新：新算法/新模型/新结构？
  ├── 应用创新：已知方法用于新场景？
  ├── 数据创新：新数据集/新实验发现？
  └── 组合创新：A+B 方法首次联合？
```

**三视角聚焦检查：**

| 视角 | 必答问题 |
|------|----------|
| 导师视角 | 这个题目是否适合当前阶段？能否形成毕业/开题/投稿所需的论文链条？ |
| 审稿人视角 | 与近三年工作的本质区别是什么？贡献是否可被一句话说清？ |
| 执行视角 | 数据、实验、仿真、代码、设备、时间是否可控？最小可行研究版本是什么？ |

**聚焦输出必须包含：**

- 研究对象：研究谁/什么系统/什么材料/什么场景
- 研究问题候选：至少 1 个主问题 + 2-3 个子问题
- 核心问题：要解释、预测、优化或验证什么
- 方法路线：理论/仿真/实验/算法/综述/系统实现
- 评价指标：至少 3 个可检索、可验证或可实验测量的指标
- 创新候选：方法创新/应用创新/数据创新/组合创新中的 1-2 类
- 最小可行版本：时间紧张时仍能完成的降级研究方案

### 1.4. 选题预审

> **目的：** 用审稿人视角审视选题——如果选题有硬伤，后面 7 步都是白做。

**基础三问预审框架：**

| 维度 | 核心问题 | 评估 |
|------|---------|:---:|
| **Originality** | 这个方向最近 3 年被大量发表了吗？你的切入点与已有工作的本质区别是什么？ | 🟢/🟡/🔴 |
| **Importance** | 解决了什么实际问题？谁会在乎这个结果？ | 🟢/🟡/🔴 |
| **Feasibility** | 数据/实验条件你能获取吗？文献量是否足够？方法论是否在可控范围内？ | 🟢/🟡/🔴 |

**评分制预审：** 红黄绿必须附带 0-5 分和一句证据理由。

| 维度 | 评分问题 | 0-1 | 2-3 | 4-5 |
|------|----------|-----|-----|-----|
| Originality | 是否有清晰差异化贡献？ | 基本重复 | 有局部差异 | 差异明确且可验证 |
| Importance | 问题是否值得解决？ | 价值不清 | 有应用价值 | 学术/工程价值明确 |
| Feasibility | 条件是否可控？ | 缺数据/设备/方法 | 有风险但可调整 | 资源和路径清楚 |
| Literature support | 文献支撑是否足够？ | 过少或过散 | 可支撑但需扩展 | 有足够核心文献链 |
| Method readiness | 方法是否能落地？ | 方法不匹配 | 需简化或替代 | 路线成熟且可执行 |

**总分判定：**

| 总分 | 信号 | 建议 |
|------|:----:|------|
| 21-25 | 🟢 绿灯 | 继续 Step 2 |
| 15-20 | 🟡 黄灯 | 标注风险后继续，或回 1.3 调整 |
| <15 | 🔴 红灯 | 回 1.2 换方向或重构题目 |

**预审结论：**

| 信号 | 含义 | 建议 |
|:----:|------|------|
| 🟢 绿灯（3 绿） | 选题可行 | 继续 Step 2 |
| 🟡 黄灯（1-2 红） | 有风险，需调整 | 回到 1.3 修正切入点 |
| 🔴 红灯（3 红） | 选题有硬伤 | 回到 1.2 换方向 |

**新增反方挑战区块：**

在五维评分之外，必须补一个“最强反对意见”区块，至少回答：

- `devils_advocate_challenge`：如果我是最苛刻的审稿人，我会如何攻击这个题目？
- `fatal_risks`：这个题最可能失败的 1-3 个原因是什么？
- `what_would_fail_this_topic`：如果后续检索或证据不支持，哪条核心前提会先塌？

### 1.5. 检索深度自动推断 🆕

> **目的：** 基于用户画像和选题预审结果，自动推断后续文献检索的深度层级，无需额外用户输入。
> 推断结果写入 `研究主题.md` 的 YAML 元数据区，供 Step 3 和 Step 4 读取。

**推断规则：**

| 用户阶段 | 时间压力 | 选题类型 | 推断 Tier | 参数 |
|---------|---------|---------|:---------:|------|
| 博一/找方向 | 无紧迫 | 探索型 | **Standard** | limit=50, 3策略 |
| 博二/开题 | 6个月 | 聚焦型 | **Standard** | limit=50, 3策略 |
| 博三+/冲刺 | 3个月投稿 | 确定型 | **Quick** | limit=30, 单策略 |
| 综述写作 | 无紧迫/6个月 | 综述型 | **Deep** | limit=100, 3策略+seminal+review |
| 教师/研究员 | 3-6个月 | 聚焦型 | **Standard** | limit=50, 3策略 |
| 工程师/快速调研 | 紧迫 | 探索型 | **Quick** | limit=30, 单策略 |
| 未明确提及阶段 | 未知 | 未知 | **Standard**（默认） | limit=50, 3策略 |

**输出**：在 `研究主题.md` 末尾写入：
```yaml
---
tier: quick|standard|deep
tier_reason: "博三冲刺+3个月投稿 → Quick 快速扫描"
methodology_blueprint:
  paradigm: "positivist|interpretivist|pragmatist|hybrid"
  method_family: "review|experiment|simulation|survey|case|mixed"
  data_strategy: "primary|secondary|both"
  evidence_priority: ["review", "experiment", "data", "standard"]
---
```

### 1.7. Step 2/3 Handoff Block

## 9. 故障排除 (Troubleshooting)

- **用户说不清方向**：不要机械问卷，用 1.2 的发散方法帮用户看到更多可能
- **预检索结果太少**：放宽年限、扩大同义词、尝试不同数据源
- **文献太多/方向拥挤**：收窄对象、场景、方法、指标或数据类型，保留 1 个差异化切入点
- **文献太少/方向过窄**：扩展上位概念、同义词和邻近应用，必要时改为综述或探索型选题
- **创新不足**：优先尝试“新场景 + 成熟方法”或“成熟问题 + 新指标/新数据”的组合创新
- **可行性不足**：降低实验强度，转为仿真、公开数据、系统评价或小样本验证
- **时间太紧**：切换 Quick tier，给出最小可行研究版本
- **预审红灯**：不要强行推进，诚实告知用户选题风险，建议换方向或重构题目
