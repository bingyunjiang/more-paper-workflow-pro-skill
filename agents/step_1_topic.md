> 旧版 Step 1 仅靠 5 轮问答让用户自述研究方向。v2.0 在对话中注入外部信息（文献量/趋势/gap/创新性），让用户做有依据的决策。

执行本步骤前，必须确认以下文件已加载：

- [ ] `.skill-state/error_log.md` — 已知错误及修复规则
- [ ] `.skill-state/decision_log.md` — 影响本 Step 的结构性决策

## 适用任务 (Applicable Tasks)

- 新用户首次使用，需要厘清研究方向
- 已有大致方向但需要精确到具体选题
- 选题可行性评估（文献饱和度、gap 识别、创新点预判）
- 研究阶段诊断（硕士/博士/青年教师 → 推荐最优入口）

---

## 不适用任务 (Non-applicable Tasks)

以下任务不属于本 Step 范围：

- 已有清晰选题和文献 → 可直接跳到 Step 3（检索方案）或 Step 5（下载）
- 大纲生成 → 路由到 `agents/step_2_outline.md`
- 文献检索 → 路由到 `agents/step_4_search_score.md`

> **硬边界：** Step 1 只生成问题结构，不生成章节结构。一级/二级标题、章节顺序、章节证据需求表完整版都属于 Step 2。

|------|------|------|:--:|

**Direct-entry input contract：**

Step 1 本身就是独立入口，只要求用户提供研究方向或待澄清的问题。用户身份、阶段、方法偏好或既有材料缺失时，应在当前 Step 内通过决策包补足最小依据，不要求用户先完成任何其他 Step。

**中文论文库提前登录：** 一旦用户研究方向、目标语料或关键词包含中文，且 Step 1 计划使用 CNKI/万方预检索，应在第一轮互动时提醒用户提前完成两站登录，不要等到候选题已经生成后才触发登录门。若需要 CDP 浏览器，Agent 直接运行：

```bash
python3 scripts/batch_chinese_search.py --open-login-tabs --port 9223
```

该命令必须自动启动或复用持久化 CDP 浏览器，同时打开 CNKI 与万方页面，然后立即返回，让选题互动继续进行。向用户说明：“请在已打开的 CNKI、万方页面完成机构账号/CARSI 登录；我会继续收集研究对象、方法和约束，后续预检索复用该会话。”登录未完成不阻塞 Step 1 的问题澄清和候选池生成，但在真实中文预检索前必须确认登录状态或降级为 OpenAlex/Crossref 中英文映射检索。

### 决策包式交互规则 (Decision Packages)

Step 1 的交互目标不是“一次只问一个问题”，而是**一轮只收敛一个决策包**。

这里的“收敛”只指当前轮的执行范围，不是把研究想象、备选路线或反例候选直接排除。Step 1 默认采用“发散层 / 收敛层”两层结构：先把主题、假设、路线、反例放进候选池，再在当前轮把一个决策包收敛到可检索、可继续推进的状态。

默认按以下 3 个决策包推进：

1. `研究对象 + 应用场景`
2. `核心问题 + 方法路径`
3. `可行性 + 创新点边界`

执行要求：

- 首轮最多请求 4 个短答案：`论文目标/类型`、`大致方向或研究对象`、`截止时间`、`已有材料或可用资源`。允许用户用一句话、选项编号或直接上传材料回答。
- 后续每轮最多新增 2 个问题，只追问会改变题目范围、方法可行性或数据/伦理边界的缺口。
- 身份、年级、目标期刊、方法偏好等能从用户表述、附件或上下文可靠推断时，直接给出“我的理解”并允许用户纠正，不重复询问。
- 非关键缺口使用保守默认值并记录 `assumed`，不要为了填满 YAML 继续盘问。
- 一轮结束必须给出一个中间结论，不能只继续追问。
- 不要一轮同时推进多个互不相关的不确定性。
- 当信息足够进入 Step 2 时，主动停止澄清并切换到产出模式。
- 若用户在侧边聊天或补充对话中提出新的探索点，先作为候选池内容记录，不要立刻并入正文或强行替换当前决策包。
- 这里收口的是当前轮执行范围，不是锁死研究想象，不锁死研究想象；新方向先记候选，后续可以再收敛。

**回答负担控制：**

- 不发送完整问卷，不要求用户逐字段填写 YAML。
- 每次提问前先复述已知信息，只列仍需用户决定的 1-2 项。
- 给出 2-3 个可选答案和推荐项；用户也可自由补充。
- 用户回答“都可以/不确定”时，由 Agent 基于目标、时间、资源和预检索提出默认方案，不重复同一问题。
- 用户上传开题报告、大纲、数据说明、项目文档或代表论文后，先抽取信息并列出推断结果，只询问无法安全推断的关键项。

中间结论至少应包括以下之一：
- 当前聚焦对象
- 当前问题定义
- 候选方向取舍
- 是否进入 Step 2

## 执行流程 (Execution Flow)

### 执行流程总览

```
Step 1.1: 研究阶段诊断      → 用户画像 + 推荐入口 + research_intent_type
Step 1.2: 广度探索 + 预检索  → 发散子方向 → 文献量/趋势/gap初判 + Socratic 澄清
Step 1.3: 深度聚焦          → 研究问题候选 + 方法论深化 + 创新点预判
Step 1.4: 选题预审          → originality/importance/feasibility + 证据支撑评分 + 反方挑战
                              🟢绿灯 / 🟡黄灯 / 🔴红灯
Step 1.5: 检索深度推断      → quick|standard|deep + 参数理由 + 方法蓝图交接

产出: 研究主题.md（增强版）
  含结构化元数据 + 用户画像 + 预检索证据 + 研究问题候选 + scope 边界 + 方法蓝图 + 创新点初判 + 预审结论 + Step 2/3 handoff
```

### 1.1. 研究阶段诊断

> **目的：** 先了解用户是谁、处于什么阶段，再决定走哪条路。不是所有人都需要完整的 8 步流程。

**诊断维度：**

| 诊断项 | 选项 | 推荐入口 |
|--------|------|---------|
| **身份** | 硕士生 / 博士生 / 研究者 / 工程师 | — |
| **年级** | 博一（找方向）/ 博二（开题）/ 博三+（冲刺） | — |
| **当前阶段** | 刚入学，需要导师给方向 | → 先泛读综述，再走 Step 1-8 |
| | 已有大致方向，需精确选题 | → 完整走 Step 1-8 |
| | 已有具体题目，需文献支撑 | → 跳 Step 3（检索方案） |
| | 已有大纲，需下载文献写论文 | → 跳 Step 5（下载） |
| | 已有初稿，需润色 | → 跳 Step 8（润色） |
| **时间压力** | 3 个月内要投稿 / 6 个月 / 无紧迫 | 影响检索年限和下载策略 |
| **目标期刊** | 已有目标 / 不确定 / 毕业就行 | 影响写作和润色标准 |

**首轮最小交互模板：**

```md
为了先给出可比较的候选题目，请用短语回答 4 项即可：
1. 目标：毕业论文 / 开题 / 期刊或会议 / 综述 / 工程研究
2. 大致方向或研究对象：一句话即可
3. 截止时间：大约多久
4. 已有条件：数据、设备、软件、项目材料、代表论文；没有可写“暂无”
```

如果用户已经提供其中任意信息，只询问剩余关键项，不重复展示完整模板。

**条件触发追问：**

| 触发条件 | 最多追问 | 不触发时处理 |
|----------|----------|--------------|
| 实证/算法题但数据来源不明 | 数据是否已有；若无，是否可用公开数据或仿真替代 | 已有数据说明则直接抽取 |
| 实验题但设备/预算不明 | 可用设备；是否接受仿真/公开数据的最小方案 | 非实验题不询问设备 |
| 医疗、人类参与或敏感数据 | 授权/伦理路径；可否改用公开脱敏数据 | 普通公开数据不询问伦理细节 |
| 企业/项目材料 | 哪些内容可公开；哪些参数必须脱敏 | 无工程材料不询问保密边界 |
| 方法明显超过期限或能力 | 是否接受简化方法或综述/案例替代 | 路线可控时不询问技能清单 |
| 导师/项目有硬约束 | 不可改变的对象、方法或成果形式 | 用户未提及且无材料证据时记为 unknown，不主动增加问题 |

只有数据/实验/授权/伦理/硬约束缺失到无法判断可行性时，暂停“最终确定题目”；仍应先给候选方向和最小可行替代方案。

**若用户方向极模糊**（如“新能源”“医学影像”“AI+教育”），不要先追问细节。先进入主动发散模式：

| 发散轴 | 输出要求 |
|--------|----------|
| 应用场景 | 列出 3-5 个可落地场景 |
| 方法路线 | 列出可做综述/仿真/实验/算法/系统实现的路线 |
| 数据或实验条件 | 判断公开数据、实验平台、仿真工具或问卷访谈是否可得 |
| 论文策略 | 给出快毕业型、稳妥开题型、高创新投稿型三类路线 |
| 风险提示 | 说明每条路线的文献支撑、创新拥挤度和执行成本 |

发散模式的目标是扩大候选池，不是提前替代定题；只有当用户选择或证据支持后，才进入收敛模式。

**Socratic 澄清输出要求：**

当用户输入属于“兴趣/模糊方向”而不是明确研究问题时，Step 1 必须在 1.1-1.3 之间逐步逼出以下结构化结果，而不是直接跳到大纲：

- `research_intent_type`：探索型 / 明确研究型 / 综述型 / 事实核查型 / 系统综述型
- `research_question_candidates`：1-3 个候选研究问题
- `scope_boundaries`
  - `in_scope`
  - `out_of_scope`
- `sub_questions`：2-3 个子问题

若用户一开始就给出清晰研究问题，可压缩该过程，直接输出结构化 RQ，不强制长对话。

**侧边聊天继承规则：**

- 对话中新增的反例、新方向、替代方法或约束条件，默认先进入候选池。
- 这些候选只在当前轮不阻塞主决策包，不代表被否决。
- 若候选后来被确认更优，再在下一轮或下一步显式替换当前主线。

### 1.2. 广度探索 + 预检索

> **目的：** 用户说出一个大致方向后，先做一次快速文献扫描——让数据说话。

执行前读取 `references/early-research-evidence-calibration.md`。Step 1 真实可调用 OpenAlex、Crossref、Semantic Scholar，以及登录后的 CNKI/万方；普通网页只作为术语、标准和机构报告补充。不得把网页搜索摘要或 API 截断样本当作完整论文库。

中文路线进入本节前，先执行输入要求中的 `--open-login-tabs`；若 CDP 启动失败，立即给出浏览器路径/端口诊断，并继续非中文源预检索，不把浏览器故障变成 Step 1 入口锁。

**流程：**

```
用户说"充电桩热管理" →
  ① 发散 3-5 个子方向：
      ├── 冷板式液冷散热
      ├── 相变材料（PCM）热缓冲
      ├── 充电枪接口发热
      ├── 功率模块 IGBT 温升
      └── 整柜风道优化
  ② 逐个子方向快速预检索（Semantic Scholar / Crossref）
  ③ 趋势判断：
      相关率高 + 多源一致 → 文献基础较稳定，继续检查拥挤度与差异化
      相关率中等或查询敏感 → 调整对象/方法/场景/指标组合
      相关率低或单源命中 → 扩展同义词并保留“待 Step 4 复核”
  ④ 输出对比表，让用户选择 1-2 个进入深度聚焦
```

**预检索执行协议：**

| 项目 | 要求 |
|------|------|
| 子方向数量 | 3-5 个，必须覆盖不同对象/方法/场景，不要只是同义改写 |
| 关键词组 | 每个子方向至少 2 组关键词：中文方向需中英双语，英文方向至少包含同义词/缩写 |
| 数据源 | 优先 Crossref/OpenAlex/Semantic Scholar；中文方向可补 CNKI/万方入口提示 |
| 检索规模 | 每个子方向观察前 20-50 条去重样本；只有数据源明确返回并保存 `total_hits` 时才可报告总量 |
| 时间窗口 | 默认关注近 3 年趋势，同时保留经典/综述线索 |
| 证据记录 | 至少记录 `observed_count / relevance_ratio / recent_ratio / source_agreement / query_sensitivity`，以及可回查代表论文/综述 2-3 篇 |
| 降级策略 | 外部检索失败时，明确标注“需 Step 4 复核”，不得伪造文献量 |

**子方向对比表模板：**

| 子方向 | 查询组 | 观察样本/相关率 | 近三年比例 | 多源一致性 | 代表论文/综述 | 初判 gap | 查询敏感性 | 建议 |
|--------|--------|-----------------|------------|------------|---------------|----------|------------|------|
| 方向 A | 2+ query sets | n / ratio | ratio | 高/中/低 | 2-3 条 DOI/source_id | 1-2 条 | 稳定/敏感/未知 | 保留/收窄/扩词 |

**样本解释边界：** capped API/CDP 样本只能判断召回质量、近期活动和术语稳定性，不能据此推断领域总规模或直接证明创新 gap。创新性表达必须降级为“预检索未见/需 Step 4 覆盖检索验证”。

### 1.3. 深度聚焦

> **目的：** 进入深度聚焦——不只问"做什么"，还预判"创新点在哪"。

**论证张力检查：**

深度聚焦不是把候选方向平均展开，而是先找研究张力。每个候选题目进入预审前，必须回答：

- `惊讶点`：已有理论、工程常识或主流方案预期会看到什么？用户材料或预检索线索显示了什么不同？
- `主角发现`：如果这篇论文只能保留一个核心贡献，哪个问题、现象、方法或工程矛盾是主角？
- `什么可以不写`：哪些子方向存在于候选池中，但不直接支撑主角发现，应降级为背景、补充或后续研究？
- `最小对比`：用一组近似题目说明为什么当前题目保留、相邻题目降级或排除。

若答不出 `惊讶点` 或 `主角发现`，不要直接宣布题目已确定；应回到 1.2 扩展候选，或把本轮结果标为可继续版本。

**双循环结构：**

```
Cycle 1: 方法论深化
  从"仿真" → 深化为 "CFD 参数化建模 + 代理模型优化 + 实验验证"
  → 方法越具体，后续检索方案越精准

Cycle 2: 创新点预判
  基于该方向的 gap 初判，与用户讨论可能的创新切入：
  ├── 方法创新：新算法/新模型/新结构？
  ├── 应用创新：已知方法用于新场景？
  ├── 数据创新：新数据集/新实验发现？
  └── 组合创新：A+B 方法首次联合？
```

**三视角聚焦检查：**

| 视角 | 必答问题 |
|------|----------|
| 导师视角 | 这个题目是否适合当前阶段？能否形成毕业/开题/投稿所需的论文链条？ |
| 审稿人视角 | 与近三年工作的本质区别是什么？贡献是否可被一句话说清？ |
| 执行视角 | 数据、实验、仿真、代码、设备、时间是否可控？最小可行研究版本是什么？ |

**聚焦输出必须包含：**

- 研究对象：研究谁/什么系统/什么材料/什么场景
- 研究问题候选：至少 1 个主问题 + 2-3 个子问题
- 核心问题：要解释、预测、优化或验证什么
- 方法路线：理论/仿真/实验/算法/综述/系统实现
- 评价指标：至少 3 个可检索、可验证或可实验测量的指标
- 创新候选：方法创新/应用创新/数据创新/组合创新中的 1-2 类
- 最小可行版本：时间紧张时仍能完成的降级研究方案

### 1.4. 选题预审

> **目的：** 用审稿人视角审视选题——如果选题有硬伤，后面 7 步都是白做。

**基础三问预审框架：**

| 维度 | 核心问题 | 评估 |
|------|---------|:---:|
| **Originality** | 这个方向最近 3 年被大量发表了吗？你的切入点与已有工作的本质区别是什么？ | 🟢/🟡/🔴 |
| **Importance** | 解决了什么实际问题？谁会在乎这个结果？ | 🟢/🟡/🔴 |
| **Feasibility** | 数据/实验条件你能获取吗？文献量是否足够？方法论是否在可控范围内？ | 🟢/🟡/🔴 |

**评分制预审：** 红黄绿必须附带 0-5 分和一句证据理由。

| 维度 | 评分问题 | 0-1 | 2-3 | 4-5 |
|------|----------|-----|-----|-----|
| Originality | 是否有清晰差异化贡献？ | 基本重复 | 有局部差异 | 差异明确且可验证 |
| Importance | 问题是否值得解决？ | 价值不清 | 有应用价值 | 学术/工程价值明确 |
| Feasibility | 条件是否可控？ | 缺数据/设备/方法 | 有风险但可调整 | 资源和路径清楚 |
| Literature support | 文献支撑是否足够？ | 过少或过散 | 可支撑但需扩展 | 有足够核心文献链 |
| Method readiness | 方法是否能落地？ | 方法不匹配 | 需简化或替代 | 路线成熟且可执行 |

**总分判定：**

| 总分 | 信号 | 建议 |
|------|:----:|------|
| 21-25 | 🟢 绿灯 | 继续 Step 2 |
| 15-20 | 🟡 黄灯 | 标注风险后继续，或回 1.3 调整 |
| <15 | 🔴 红灯 | 回 1.2 换方向或重构题目 |

**预审结论：**

| 信号 | 含义 | 建议 |
|:----:|------|------|
| 🟢 绿灯（3 绿） | 选题可行 | 继续 Step 2 |
| 🟡 黄灯（1-2 红） | 有风险，需调整 | 回到 1.3 修正切入点 |
| 🔴 红灯（3 红） | 选题有硬伤 | 回到 1.2 换方向 |

**新增反方挑战区块：**

在五维评分之外，必须补一个“最强反对意见”区块，至少回答：

- `devils_advocate_challenge`：如果我是最苛刻的审稿人，我会如何攻击这个题目？
- `fatal_risks`：这个题最可能失败的 1-3 个原因是什么？
- `what_would_fail_this_topic`：如果后续检索或证据不支持，哪条核心前提会先塌？

## 故障排除 (Troubleshooting)

- **用户说不清方向**：不要机械问卷，用 1.2 的发散方法帮用户看到更多可能
- **预检索结果太少**：放宽年限、扩大同义词、尝试不同数据源
- **文献太多/方向拥挤**：收窄对象、场景、方法、指标或数据类型，保留 1 个差异化切入点
- **文献太少/方向过窄**：扩展上位概念、同义词和邻近应用，必要时改为综述或探索型选题
- **创新不足**：优先尝试“新场景 + 成熟方法”或“成熟问题 + 新指标/新数据”的组合创新
- **可行性不足**：降低实验强度，转为仿真、公开数据、系统评价或小样本验证
- **时间太紧**：切换 Quick tier，给出最小可行研究版本
- **预审红灯**：不要强行推进，诚实告知用户选题风险，建议换方向或重构题目
